# 04 Code and Repository

## 代码修改原则
- 先理解现有代码和规则，再修改。
- 优先小步、可验证、可回滚的改动。
- 不做与任务无关的重构。
- 不隐藏错误，不降低验证强度。
- Python 代码应清晰、模块化，有必要的类型提示。
- 复杂逻辑用简洁中文注释解释原因，而不是复述代码。
- 配置优先于硬编码。

## 代码规范

在src/内，分模块放你的代码，最好形成模块树。每一个代码模块必须有README作为索引。

在写代码之前，你需要检查：
- 有没有现有的类似代码？是否一定要写新代码？能否通过简单的修改就可复用现有代码？
- 你需要优先保证代码足够干净，兼容旧数据不是强制要求；不要为了复用而复用，你应该保持整体代码的可读性

## Git worktree
在本地修改代码时，你应该使用../nimloth-<branch-name> （分支名的`/`在文件夹名称内使用`-`），不要直接在main分支修改，除非Prompt里有显式说明。

## 文件保护

禁止修改：

- `ai_notes/archive/` 下内容；
- `qc_*.md`；
- 人类明确标记为只读的文件；
- 未经确认的大型数据、模型权重和实验输出。

如必须修改保护内容，先提出原因并等待批准。

## Git 规则
- 开始前检查 `git status --short --branch`。
- 不要覆盖他人未提交改动。
- 阶段性成果应提交，提交前说明修改范围和验证情况。
- 禁止工作完毕后不整理工作区。
- 不在不清楚分支策略时创建/切换/合并重要分支。
- 禁止直接在main分支修改
- 使用Semi-linear merge合并策略
