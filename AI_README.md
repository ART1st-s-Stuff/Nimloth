## 开发行为准则
你是本项目的AI开发助手，本项目的人类开发者一直以来非常重视你所做出的贡献。以下是你需要严格遵循的行为准则，与人类开发者进行有效的互动。

prompt/代码中没有清楚描述/你不理解的部分，征询人类开发者意见。
遇到可选超过一种方案的情况，征询人类开发者意见。
如果要进行大范围改动，征询人类开发者意见。
如果要进行破坏性改动，征询人类开发者意见。

善用子Agent，将合适的任务分配给子agent，减少不重要的中间信息对Root agent上下文的占用。Root agent的上下文始终应该保存重要信息。
善用命令行和python。善用正则表达式搜索。

本文件由人类撰写，如有更改需求需要提出issue（见下文）。

### 目录规范
【注意！】本文档所在目录就是项目根目录（通常名为flower），所有脚本都应假定为在此目录下执行。在实际的任务执行session中，你可能会读取到当前的根目录是某个上层目录，不要认为脚本会在上层目录执行。
- AI_README.md: 本文档
- src/: 源代码目录 【为你的源代码分模块】
- configs/: 配置文件目录
- scripts/: 脚本目录
- models/: 保存模型的目录 【根据训练阶段分类保存】
- datasets/: 保存数据集的目录 【根据训练阶段分类保存】
- 【如果你需要git submodules，可以添加在主目录也可以添加到src】


### 关于任务
对于短的、局部性的任务，人类会直接将要求写在prompt中。
对于长的、系统性的任务，人类会在ai_tasks中先编写任务说明。你需要在ai_tasks/ai_progress/中【实时地】保存任务的执行进度、规范等信息（如果是永久性的规范，则应当写入代码或notes）。如果当前任务的说明与实际代码有冲突，或者开发者要求，你可以修改ai_tasks中的当前任务文件。但非必要不应当对任务说明文件进行改动。

### 代码要求
- 尽量复用现有代码
- 创建新代码时需要具备可复用性和可配置性，保证后续可持续发展
- 使用Hydra保存配置，尽量不要在shell里覆盖
- 保持代码逻辑简洁且易于理解
- 函数、变量的命名也要易于理解
- 遵循统一的命名规范。你可以将代码规范写在笔记里。

### 文件要求
- 遵循项目结构
- 模块化，按职责分离

### 注释要求
- 使用中文
- 遵循Python规范
- 简洁易懂，面向人类开发者
- 复杂逻辑额外添加注释说明

### AI笔记
- 由于Agent不可能看到所有聊天历史记录，所以你需要在ai_notes/内保存经常使用的关键信息，做好索引和分类。note的格式、内容没有具体要求，你可以按需记录短的句子或长的内容，只要你认为信息对开发有所帮助。*建议制定一套标准格式，便于使用脚本*
- 尽量节约Token。例如对于简单阅读代码已经完全足够的情况，你可以保存信息的索引，甚至无需记录。另外，由于人类也会频繁更改代码，而且人类不会更新ai_notes，所以你不应该在你的笔记中包含【具体】的代码逻辑。
- 你可以执行命令在ai_notes/中搜索。
- 禁止包含用户个人信息。
- 禁止包含和当前运行环境有关的信息。
- 包含一个简洁的main.md作为阅读的入口。
- 在任务执行过程中，你应当自行判断合适的时机添加笔记，不要等人类给你下达添加笔记的指令。

### AI笔记的更新
- 对于每条note，记录重要性、一个整数L值（初始化为1）和D值（dev date）。
- dev date遵循以下约定：
  - 值为一个整数，使用ai_notes/today.txt获取
  - ai_notes/today.txt的格式如下，指2026年4月8日对应dev date为1
    ```
    2026-04-08 1
    ```
  - 如果你发现今天的日期晚于today.txt的日期，则修改dev.txt将dev date加一
- 根据以下原则更新信息：
  - L值和dev date
    - 如果某个note对完成任务有重要帮助，将L值加一，并更新dev date
    - 如果某个note对完成任务没有帮助，则不要更新L值和dev date
  - 更新笔记内容
    - 如果你使用了某条笔记，不论是否对任务有帮助，如果内容和当前代码不符，那么你需要根据实际情况进行修正或删除。如果对当前任务没有帮助，即使更新了内容，也【不要】更新L值和日期。
  - 重要性可以根据实际情况调整
- 关于重要性和时效性的原则
  - 具体要求保存在ai_notes/update_rules.md中。【非必要不更新】
- 提示：完成任务（或大任务的一个阶段）后，使用子Agent来执行上述更新。
