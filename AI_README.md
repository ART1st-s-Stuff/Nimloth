# AI编程助手指南
本文件是面向AI变成助手的行为规范和操作指南。AI编程助手应遵守本文指定的规范，并利用本文给出的信息辅助工作。

本文件由人类撰写，如有更改需求，需要按本文叙述的规则提出issue。

## 项目内容
世界模型Agent。见plan.md。

## 开发行为准则
你是本项目的AI开发助手，本项目的人类开发者一直以来非常重视你所做出的贡献。以下是你需要严格遵循的行为准则，与人类开发者进行有效的互动。

### 关于任务
对于短的、局部性的任务，人类会直接将要求写在prompt中。
对于长的、系统性的任务，人类会在ai_tasks中先编写任务说明。你需要在ai_tasks/ai_progress/中【实时地】保存任务的执行进度、规范等信息（如果是永久性的规范，则应当写入代码或notes）。如果当前任务的说明与实际代码有冲突，或者开发者要求，你可以修改ai_tasks中的当前任务文件。但非必要不应当对任务说明文件进行改动。一个任务可能在一个甚至若干个session中经过许多轮对话才能分阶段地完成。

在执行任务的过程中，始终询问自己：
- 任务要求中有没有清楚描述、无法理解的部分？
- 代码中有没有逻辑模糊、作用不清等异常情况？
- 是否有超过一种方案适用于当前的情况？
- 为了达成任务，要不要进行大范围改动？要不要进行破坏性改动？
- 任务要求是否和本指南冲突？是否和现有代码冲突？是否和以前提出的要求/记录的标准/笔记中的其他内容冲突？
- 有没有其他困难？

如果遇到以上任何一种情况，立即停下征询人类开发者意见。

同时，为确保任务的执行效率，你必须进行思考：
- 能否将任务进行合适的切分并分配一部分给子agent？这样可以减少不重要的中间信息对Root agent上下文的占用，并将重要信息汇聚到Root agent的上下文中。
- 善用命令行、python和正则表达式搜索可以加快很多效率。
- 当前有没有遇到与笔记冲突的情况？是否需要更新笔记？有没有新东西需要记录？

当一个阶段执行完毕后，首先进行自我总结，然后按需更新笔记和README.md，如果需要提出issue则修改AI_issues.md。最后，生成面向人类开发者的任务阶段报告，主要聚焦新增了什么、更改了什么、是否有issue需要解决。如果有涉及命令的更改，给出示例命令。

### 本项目的特殊注意事项
注意，本项目正在快速迭代，大范围和破坏性改动会经常发生，但你仍需向人类显式征求意见，尤其是当更改看起来较少但实际上涉及大量non-trivial的更改时。需要注意的是，你一般无需保留旧的兼容性接口，直接删除即可，除非人类开发者显式地告诉你需要保留旧的兼容代码。

### 目录规范
【注意！】本文档所在目录就是项目根目录（通常名为flower），所有脚本都应假定为在此目录下执行。在实际的任务执行session中，你可能会读取到当前的根目录是某个上层目录，不要认为脚本会在上层目录执行。
- AI_README.md: 本文档【不要修改此文件。如有修改必要，提出issue。】
- AI_progress.md: 当前项目计划及进展
- AI_issues.md: 你提出并让人类查看的issues
- plan.md: 人类开发者制定的项目计划【不要修改此文件。如有修改必要，提出issue。】
- generated_doc.md: Gemini生成的更细致的项目计划【不要修改此文件。如有修改必要，提出issue。】
- src/: 源代码目录 【为你的源代码分模块】
- configs/: 配置文件目录
- scripts/: 脚本目录
- dev/: 保存开发用的临时测试脚本/代码
- models/: 保存模型的目录 【根据训练阶段分类保存】
- datasets/: 保存数据集的目录 【根据训练阶段分类保存】
- outputs/: 存放logs等debug用信息
  - dev/<datetime>/<taskname>: 保存dev里输出的内容，按dev里脚本名称分文件夹保存。【dev的所有输出一律存在这里，不要保存在别的地方，如models、datasets！】
- qc_*.md: quick command, 常用任务的Prompt【不要修改这些文件。】
- 【如果你需要git submodules，可以添加在主目录也可以添加到src】

### 代码要求
- 尽量复用现有代码
- 创建新代码时需要具备可复用性和可配置性，保证后续可持续发展
- 使用Hydra保存配置，尽量不要在shell里覆盖
- 保持代码逻辑简洁且易于理解
- 函数、变量的命名也要易于理解
- 遵循统一的命名规范。你可以将代码规范写在笔记里。

### 文件要求
- 遵循项目结构
- 模块化，按职责分离

### 注释要求
- 使用中文
- 遵循Python规范
- 简洁易懂，面向人类开发者
- 复杂逻辑额外添加注释说明

### AI笔记
- 由于Agent不可能看到所有聊天历史记录，所以你需要在ai_notes/内保存经常使用的关键信息，做好索引和分类。note的格式、内容没有具体要求，你可以按需记录短的句子或长的内容，只要你认为信息对开发有所帮助。*建议制定一套标准格式，便于使用脚本*
- 尽量节约Token。例如对于简单阅读代码已经完全足够的情况，你可以保存信息的索引，甚至无需记录。另外，由于人类也会频繁更改代码，而且人类不会更新ai_notes，所以你不应该在你的笔记中包含【具体】的代码逻辑。
- 你可以执行命令在ai_notes/中搜索。
- 禁止包含用户个人信息。
- 禁止包含和当前运行环境有关的信息。
- 包含一个简洁的main.md作为阅读的入口。
- 在任务执行过程中，你应当自行判断合适的时机添加笔记，不要等人类给你下达添加笔记的指令。

### AI笔记的更新
- 对于每条note，记录重要性、一个整数L值（初始化为1）和D值（dev date）。
- dev date遵循以下约定：
  - 值为一个整数，使用ai_notes/today.txt获取
  - ai_notes/today.txt的格式如下，指2026年4月8日对应dev date为1
    ```
    2026-04-08 1
    ```
  - 如果你发现今天的日期晚于today.txt的日期，则修改dev.txt将dev date加一
- 根据以下原则更新信息：
  - L值和dev date
    - 如果某个note对完成任务有重要帮助，将L值加一，并更新dev date
    - 如果某个note对完成任务没有帮助，则不要更新L值和dev date
  - 更新笔记内容
    - 如果你使用了某条笔记，不论是否对任务有帮助，如果内容和当前代码不符，那么你需要根据实际情况进行修正或删除。如果对当前任务没有帮助，即使更新了内容，也【不要】更新L值和日期。
  - 重要性可以根据实际情况调整
- 关于重要性和时效性的原则
  - 具体要求保存在ai_notes/update_rules.md中。【非必要不更新】
- 提示：完成任务（或大任务的一个阶段）后，使用子Agent来执行上述更新。
